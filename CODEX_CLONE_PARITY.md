# Codex Clone Parity Tracker

This tracker defines what must be true before this Jan workspace can reasonably claim to be a full Codex app/CLI clone rather than a partial bridge.

## Implemented app-server transport and session core

- App-server and proto session transports are wired through `web-app/src/lib/codex-app-server`.
- Tauri process launching supports Codex app-server stdio and proto fallback paths.
- Chat streaming maps assistant, reasoning, plan, command output, file-change, process, account, MCP, remote-control, thread lifecycle, capability-change, review, moderation, and model status events into visible UI message chunks.
- Approval, server requests, interrupt, compact, reload, rollback, review, command execution, shell/process, filesystem, MCP, account, remote control, config/admin, plugins, skills, marketplace, thread lifecycle, realtime, and raw RPC controls are exposed through the Codex capabilities panel.
- Codex CLI helpers for `login`, `logout`, `version`, `apply`, `completion`, `exec`, `help`, `raw` subcommands, and typed `app`/`mcp`/`mcp-server`/`proto`/`debug`/`app-server`/`cloud`/`remote-control` paths are exposed through the panel.
- CLI-native `review`, `doctor`, `features`, `plugin`, `cloud`, `remote-control`, and `help` actions are also exposed from the panel.

## Implemented visible control surfaces

- Review panel preserves git diff as authoritative and can start detached or advanced Codex reviews.
- Account block exposes account read/login/cancel/logout/rate-limit/usage/credits-nudge flows.
- Thread block exposes list/read/turns/turn-items/fork/archive/unarchive/unsubscribe/name/metadata/settings/goal/memory/reset/inject/terminal cleanup/realtime/review/interruption/compact/reload/rollback operations.
- Remote Control block exposes enable/disable/status/pairing/client list/client revoke operations.
- Config/Admin block exposes config read/value write/batch write/config requirements/permission profiles/collaboration modes/external agent detect/import/feedback upload/Windows sandbox setup.
- Models/Providers/Features block exposes model list/provider capabilities/experimental features/feature enablement/environment registration/user-input request.
- Raw app-server RPC block can call arbitrary app-server methods with JSON params for forward compatibility.
- Codex CLI block exposes CLI-native login/logout/version/apply/completion/exec/raw subcommand flows and convenience actions for `app-server`, `proto`, `mcp`, and `debug`, plus dedicated buttons for `review`, `doctor`, `features`, and `plugin`.
- Plugins/Marketplace/Skills block exposes plugin list/installed/read/install/uninstall/plugin skill read/marketplace add/remove/upgrade/skill config write/app list/skills/hooks.
- Runtime FS/Process block exposes filesystem read/write/list/metadata/mkdir/remove/copy/watch/unwatch, process spawn/stdin/resize/kill, and command exec/stdin/resize/terminate.
- MCP block exposes status/OAuth/config reload/resource read/tool call.

## Remaining implementation work

- Panel modularization is effectively complete for the Codex capabilities surface. `ModelToolsPanel.tsx` now imports extracted `ThreadsPanel`, `AccountPanel`, `RemoteControlPanel`, `ConfigAdminPanel`, `ModelsProvidersFeaturesPanel`, `McpPanel`, `PluginsMarketplaceTool`, `RuntimeFsProcessPanel`, `RawRpcTool`, `CodexCliPanel`, `CodexReviewPanel`, `CodexSummaryCards`, `AppServerRuntimeLogs`, `CodexCapabilitiesFooter`, and `ProtoFallbackNotice`, plus shared helpers in `model-tools-panel/shared/codex-helpers.ts`. Remaining work here is not more file splitting for its own sake; it is checking that the extracted props/actions still preserve behavior and reducing any overly wide component contracts when that directly improves v1 reliability.
- The previous `ThreadsPanel` extraction regression is fixed. The old inline `Clean terminals` action is restored via the `onCleanTerminals` prop and the `withTargetCodexThread` + `cleanCodexBackgroundTerminals` handler in `ModelToolsPanel.tsx`. Keep this covered in focused validation when touching thread actions again.
- Structured controls still need to replace the remaining raw JSON textareas where upstream schemas are stable enough to support safer editing. The app already has typed controls or field-first editors for many high-use paths, including account login/usage params, remote-control pairing params, config value writes, config batch edits, feature enablement, environment registration, thread metadata/settings/review basics, runtime spawn/command exec/PTY payloads, thread action params (with advanced JSON fallback), thread inject items (now row-first with advanced JSON fallback), skill config, raw RPC top-level params, Windows sandbox setup, external-agent import migration items, MCP schema-derived argument templates, and MCP simple-schema tool arguments with advanced JSON fallback. Recent pass: raw RPC plugin-share targets no longer require strict JSON input in the common path (line/comma list accepted and still validated back into structured params). Remaining work is focused on payloads still genuinely JSON-heavy or intentionally open-ended: complex config/admin import payloads beyond migration item lists, MCP nested/union tool schemas, and raw RPC params where no typed wrapper exists. A first pass has been completed in CLI controls where command-like args now accept tokenized values as a no-JSON path for common actions.
- Raw RPC is still serving as an escape hatch for functionality that should eventually have first-class controls. The searchable method catalog and presets are present, and high-use escape-hatch flows now have typed helpers for plugin sharing, thread review starts, fuzzy file search session start/update/stop with query/limit, and top-level param fields. Remaining parity work is to keep promoting raw RPC flows into typed UI when they become common enough that manual JSON edits would block v1 usability.
- Added an automated method-audit pass in `scripts/codex/cli/method-surface-check.mjs` (run via `yarn codex:parity:appserver:methods`) that compares Jan wrapper method usage against upstream `app-server` method discovery from an initialize+invalid-method probe. This should be re-run when upgrading Codex binaries, then used with CLI/GUI behavior checks to confirm wrappers/aliases still match upstream naming and availability.
- Proto fallback handling is implemented but still needs final product decisions and polish. The capabilities panel disables app-server-only controls and shows unsupported-state messaging in proto mode, but parity work remains to make each panel’s unsupported state consistent, clearly scoped, and intentional instead of relying on a broad disabled container plus local notes.
- Thread browsing is functionally present but still needs polish before it feels equivalent to a mature Codex control surface. Search, source filtering, sort, selection chips, selected-thread summary, turn chips, and item chips are all present. Remaining work is denser presentation, clearer hierarchy between loaded vs stored threads, and reducing snapshot-json dependence in favor of more readable summaries where practical.
- Runtime/process tooling is functionally present but still needs finishing work. Terminal-style sessions, search, copy-output, resize, overlay expansion, attach-handle, selected-session summary, kind-aware selected stdin/resize/stop actions, and process/command event accumulation are wired. Remaining work is UX polish around session lifecycle and reducing the amount of raw result JSON users need to inspect for normal workflows.
- MCP browsing is functionally present but still incomplete from a product perspective. Server/resource/tool selectors, descriptor lists, template generation, simple-schema field editing, advanced JSON fallback, and schema validation exist. Remaining work is better presentation for large or nested schemas, clearer resource metadata display, and smoother tool-call flows when schemas contain complex unions or optional nested structures.
- Marketplace/plugin/skill management is functionally present but still incomplete from a product perspective. Installed filtering, descriptor browsing, marketplace source cards, selected plugin/skill detail panes, metadata preview, direct actions, installed/available/source/skill summary counts, state-aware selected plugin install/uninstall actions, and field-first skill config editing with advanced JSON fallback are in place. Remaining work is presentation polish and reducing remaining JSON entry for plugin metadata-oriented actions.
- The capabilities summary surfaces are extracted but still mostly read-only. `Skills`, `Plugins (all / installed)`, and `Hooks` are now readable compact summary rows in `CodexSummaryCards.tsx` with installed/available counts and top-item previews. Remaining work is deciding whether v1 needs these to become richer actionable browsers or whether the existing marketplace/config surfaces are the canonical interaction points.
- The runtime logs surface has been extracted to `AppServerRuntimeLogs.tsx` and now supports stream filtering, severity filtering, search, clearing, timestamped rows, severity counts, and copying visible logs with timestamp/level/stream context. Remaining work is deciding whether v1 needs deeper diagnostics such as action links and per-session scoping.
- The capability footer is extracted to `CodexCapabilitiesFooter.tsx` and now avoids overclaiming a single monolithic surface. It explicitly states that Codex panels cover remote control, marketplace, config read/write, and collaboration modes, while Studio runtime actions and git worktrees are routed through Jan Studio/workspace controls. Remaining work here is only to keep that routing language current if those controls move.
- Validation evidence has current terminal coverage for the latest extracted panel state, including web build, focused Codex tests, parity scripts, and smoke script entries in the evidence section. Before claiming v1, rerun the same matrix after the final implementation changes and complete real interactive Tauri desktop smoke.
- The biggest remaining parity risk is interactive desktop validation. Browser preview evidence explicitly says plain browser preview is not a substitute for real Tauri desktop verification. `yarn codex:desktop:preflight` now verifies local prerequisites for that pass, including the Codex binary, app-server `--stdio`, required package scripts, Tauri config/capability files, smoke/parity scripts, and checklist coverage for approval and shutdown. Completion still requires the real desktop smoke pass from `DESKTOP_SMOKE_CHECKLIST.md`, including chat start/streaming, approval/deny flows, review behavior, runtime controls, MCP/account states, persistence, shutdown, and any panel interactions touched by the recent extractions.
